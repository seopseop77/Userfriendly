## Findings

1. **Row count, roles, and turn_seq are clean.** 2 user turns → 4 sink
   rows (each user turn drove one tool round). `user_input` /
   `tool_result` alternates as ADR-0038 dictates; `turn_seq` runs 1→4
   gap-free across the `--resume` boundary.
2. **`conversation_id` stable across `--resume`.** Single ULID for the
   whole round; the second `runner-allow.sh resume` call recovered the
   same conversation store from the workdir as expected.
3. **Cache mechanics confirm model-pinning works.**
   - row 1: cache_write=51805 (cold open)
   - row 2: cache_read=51805 ≡ row1.cache_write (intra-turn tool round)
   - row 3: cache_read=51805 (cumulative read of the cached system + opener)
   - row 4: cache_read=52681 ≡ row3.cache_read + row3.cache_write (51805+876)
   The chain is exact — same as the origin-probe Haiku round's behaviour,
   confirmed for sonnet.
4. **`system_prompt_jsonb` variation tracker fires only once.** Row 1
   has `system_prompt_jsonb IS NOT NULL`; rows 2–4 are NULL. Same system
   text across the whole round → variation tracker correctly suppresses
   the redundant writes. The view-side `system_prompt_resolved` should
   carry forward (will verify in a later round that has multi-message
   resolution).
5. **`request_jsonb` shape**: list-typed on every row.
   - user_input rows: `[{type:"text", text:"[PROBE …] …",
     cache_control:{ttl:"1h", type:"ephemeral"}}]` — `cache_control` is
     present on user-typed blocks (consistent with the ADR-0038
     observation that user input carries `cache_control` while
     framework-typed content does not).
   - tool_result rows: `[{type:"tool_result", content:"<file body>"}]` —
     no `cache_control` on the tool_result block.
6. **`response_jsonb` typed `object`** (no surprises here, just
   confirming ADR-0038's rename from `response_json` text → `response_jsonb`).
7. **`model_served = "claude-sonnet-4-6"` (no `[1m]` suffix).** Matches
   non-bug #1 from the origin probe: the suffix is Claude-Code-internal
   display only.

## Anomaly candidate — `claude-manage` segfault on `new` exit

- **t01 (`runner-allow.sh new`) exited with code 139 (SIGSEGV)** despite
  both required LLM round-trips landing 200 in `exchanges` and producing
  the expected rows in `plugin_analytics`. The crash happened in the
  agent process **after** the work was complete (otherwise rows would
  be missing).
- **t02 (`runner-allow.sh resume`) exited with code 0.** No segfault.
- Hypothesis: the segfault is specific to the `new` code path's shutdown
  / cleanup (or some first-call init artefact), not the resume path or
  the proxy/sink pipeline. **Reproducibility unknown across rounds yet
  — will track per-round.**
- Sub-session stdout/stderr is discarded by design, so the bash-level
  log only carries the kernel's `Segmentation fault: 11` message
  against the `claude-manage` invocation. No traceback recovered.
- Data flow: zero impact. Not a sink bug. **Worth recording but does
  not block the matrix.**
