## Findings

1. **Write `tool_use` payload accounts for output_tokens, not input.**
   Row 3 has `output_tokens=1753` (the entire 50-object JSON the
   model serialised into the `Write` tool_use arg). The user message
   itself was 309 chars / a short prompt. The model's emission of a
   large tool_use payload registers as `output_tokens` on the
   user_input row that *triggered* it — slightly counter-intuitive
   labelling (it's user_input role, but most of its token cost is
   model output). Worth noting for token-accounting queries.
2. **Write `tool_result` is small + uniform** ("File created
   successfully at: <path> (file state is current in your context — no
   need to Read it back)"). No `is_error` field.

## Anomaly candidate — UUID re-use under `new` mode

- A misfire on my side: I initially called `runner-allow.sh new <r005
  uuid> [PROBE r005 t02]` — re-using r005 t01's UUID with `new` mode
  instead of `resume`.
- `runner-allow.sh` exit code: **1** (immediate, sub-second).
- supabase: **0 rows** for `r005 t02` under that attempt; no exchange,
  no audit log entry. Pipeline cleanly rejected.
- Behaviour is **sane** — `claude` (or `claude-manage`) refuses to
  start a fresh `--session-id` when that UUID already has a
  conversation store in the workdir. **Not a bug.** Recording so the
  next probe author knows the error class (and so the runner could
  optionally emit a clearer message in the future — minor polish, not
  needed for this campaign).
