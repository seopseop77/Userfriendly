## Interpretation

The user supplied a complete decision packet up front:

1. Collection scope — full L3 storage; plugin-level toggle
   (`LLMTRACK_PLUGINS_DISABLED`) is the operator off-switch.
2. Disclosure — documentation only at this stage; no per-task consent UI.
3. Retention — 6 months, stated as policy (auto-deletion deferred).
4. Deletion requests — operator-handled SQL on `org_id` /
   `session_id`; no API surface until `session_id` is real.
5. PII scrubbing — implement; cover `sk-`, `lts_`, `Bearer <value>`,
   email addresses.
6. Scrubbing location — `HookContext` accessor (`request_text`,
   `response_content_json`); raw `_raw_request_body` and
   `_parsed_response` left untouched.

My job was to implement (5) + (6) cleanly and write the ADR + docs in a
way that doesn't paint over the trade-offs the user already settled on.
Two interpretation calls came up in flight:

- **The `llm_tracker.scrubbers` module CLAUDE.md §7 describes does not
  exist.** Phase 1c was paused before that module landed. The only
  scrubbing code in the repo is `llm_tracker_server.proxy.credential`
  (structlog event-dict redactor for Anthropic credentials). Decision:
  create a new `llm_tracker_sdk.scrubbers` module — the accessor lives in
  the SDK, so the scrubber must too to avoid inverting the package
  dependency. The credential module stays a separate defence-in-depth
  layer on the log side.
- **The ADR number in the prompt's "Plan" was `0002`, but `0002` is
  taken** (`docs/decisions/0002-task-scope-enforcement.md`). The "#2"
  refers to the Phase 3a queue item index (per the STATUS.md table),
  not the ADR number — like ADR-0020 / ADR-0026 / ADR-0027, this gets
  the next sequential number. Decision: ADR-0029.

## Decisions

- **Scrubber lives in the SDK, not the server.** The accessor calling
  it is in the SDK; importing server code into the SDK would invert the
  package dependency. The server's existing `proxy.credential` scrubber
  stays as defence-in-depth for log event dicts; the two layers don't
  share code today (different shapes — dict-walker vs. string-regex).
  Unifying them is queued if real traffic shows duplication-worth-fixing.

(All "(commit `<code>`)" tags above resolve to **`a4c08b3`** — see
§Handoff for the rolled-up commit list.)
- **Replacement tags are kind-tagged, not generic.** `[REDACTED:secret]`
  / `[REDACTED:token]` / `[REDACTED:bearer]` / `[REDACTED:email]` so an
  operator can grep historical rows for what fired. Plain `[REDACTED]`
  was the simpler alternative; the audit-grep use case won.
- **Bearer rule runs first.** A `Bearer sk-…` span becomes one tag, not
  `Bearer [REDACTED:secret]`. The unit test
  `test_bearer_consumes_sk_value_inside` pins the ordering.
- **Word-boundary over-redaction is intentional.** `\bsk-` treats `-` as
  non-word, so `task-sk-abcdefgh` over-redacts. That's a privacy-tilted
  trade-off the user's framing explicitly accepts. Pinned by
  `test_sk_after_dash_redacts_privacy_tilted` so a future cleanup
  doesn't silently invert the behaviour.
- **Storage stays canonical.** ADR-0028's "faithful reassembly" contract
  on `response_json` is not weakened by ADR-0029. The accessor is the
  privacy floor; the database row is the operator's incident-response
  surface. `docs/plugins.md` §3.2 documents the SQL-vs-accessor
  asymmetry explicitly so plugin authors who later query the table
  directly aren't surprised.
- **ADR number is 0029, not 0002.** "ADR-#2" in STATUS.md refers to the
  Phase 3a queue item index; the ADR file gets the next sequential
  number like ADR-0020/0026/0027.

## What's left / known limits

- **Operator-handled deletion only.** No API endpoint yet — landing it
  requires a real `session_id` populator (currently hardcoded
  `"server"` in the forwarder). Queued behind Phase 3b agent identity
  work per ADR-0029 Axis 4.
- **No automated 6-month retention job.** Stated as policy in ADR-0029
  Axis 3; a periodic `DELETE … WHERE started_at < now() - interval '6
  months'` is a future CP.
- **`messages_json` SQL surface is not scrubbed.** The
  `analytics_sink` plugin parses the request body directly into
  `messages_json` rather than reading through `ctx.request_text()`, so
  rows in `public.plugin_analytics` carry the canonical body. The
  privacy floor is the plugin-API boundary, not the SQL boundary —
  documented in `docs/plugins.md` §3.2 and noted in ADR-0029 §"Open
  questions". A future ADR can decide whether to extend scrubbing to
  the SQL mirror or to lean further into the retention policy.
- **i18n email addresses not redacted.** RFC 5322 ASCII subset only.
  ADR-0029 §"Open questions" notes the trigger to extend.
- **Pre-SSE upstream-failure-path row write** (ADR-0027 axis 2) and
  **`exchanges.tool_call_count` fate** remain queued from prior
  closures — unrelated to this CP.
